## 아키텍처 (Architecture)

DeepSeek‑OCR 은 **Vision‑Text 압축**을 목표로 설계된 최초의 엔드‑투‑엔드 VLM(Visual‑Language Model)이다.  
전체 구조는 **Encoder‑Decoder** 형태이며, 핵심인 **DeepEncoder**와 **DeepSeek‑3B‑MoE‑A570M** 디코더로 구성된다.  

### 1. 전체 파이프라인
```
입력 이미지 → DeepEncoder → (압축된 Vision Token) → DeepSeek‑3B‑MoE Decoder → 텍스트 출력
```
* **DeepEncoder** : 고해상도 이미지를 저활성화 메모리와 적은 Vision Token 수로 변환.  
* **DeepSeek‑3B‑MoE** : 3 B 파라미터 규모(MoE) 디코더, 실제 활성화 파라미터는 약 **570 M**.  
* **Prompt** 를 통해 OCR, 차트·화학식·기하학 파싱 등 다양한 태스크를 하나의 모델로 수행한다.  

> 자세한 전체 흐름은 논문의 Figure 3을 참고한다.  

### 2. DeepEncoder

| 구성 요소 | 파라미터 | 핵심 기능 | 특징 |
|----------|----------|-----------|------|
| **SAM‑base** (patch‑size 16) | 80 M | 윈도우‑Attention 기반 로컬 특징 추출 | 고해상도 입력에서도 활성화 메모리 낮음 |
| **16× 토큰 압축기** (2‑layer Conv) | – | `kernel=3, stride=2, padding=1`, 채널 256→1024 | 윈도우‑Attention 결과를 1/16 토큰 수로 다운샘플 |
| **CLIP‑large** (patch‑size 16, 제거된 첫 번째 patch‑embedding) | 300 M | 전역‑Dense Attention 기반 전역 지식 추출 | Vision‑Token → 텍스트 토큰 복원에 필요한 풍부한 표현 제공 |

#### 2.1 토큰 흐름
* 1024 × 1024 입력 → 4096(=1024/16 × 1024/16) 패치 토큰 생성  
* 윈도우‑Attention 처리 → 4096 토큰을 **16× 압축기** 통과 → 256 토큰(≈1/16)으로 축소  
* 축소된 토큰이 CLIP‑large 로 전달되어 전역 컨텍스트를 학습하고, 최종 **Vision Token** 으로 출력  

#### 2.2 다중 해상도 지원
DeepEncoder 은 **동적 해상도 모드**와 **고정 해상도 모드**를 모두 제공한다.

| 모드 | Native Resolution | 토큰 수 (예시) | 전처리 방식 |
|------|-------------------|----------------|--------------|
| Tiny | 512 × 512 | 64 | 이미지 **리사이즈** |
| Small | 640 × 640 | 100 | 이미지 **리사이즈** |
| Base | 1024 × 1024 | 256 | **패딩** 후 입력 |
| Large | 1280 × 1280 | 400 | **패딩** 후 입력 |
| Gundam |  n × 640 × 640 tiles + 1024 × 1024 global view | `n·100 + 256` (n∈[2,9]) | 타일링 + 패딩 |
| Gundam‑M (master) | 1024 × 1024 tiles + 1280 × 1280 global view | `n·256 + 400` | 연속 학습을 통해 확장 |

* **동적 포지셔널 인코딩 보간**을 사용해 다양한 해상도에서도 동일 모델이 동작하도록 설계.  
* `Valid Vision Tokens` 은 실제 유효 토큰 수를 의미하며, `N_valid = ceil(N_actual × [1‑((max(w,h)‑min(w,h))/max(w,h))])` 로 계산한다(식 (1) 참고).  

### 3. MoE Decoder (DeepSeek‑3B‑MoE‑A570M)

* **Mixture‑of‑Experts (MoE)** 구조 – 64개의 라우팅 전문가 중 6개와 2개의 공유 전문가가 활성화된다.  
* 총 파라미터 **3 B**, 활성 파라미터 **570 M** → 500 M‑급 모델 수준의 추론 속도와 3 B 모델 수준의 표현력을 동시 제공.  
* 입력은 압축된 Vision Token `Z ∈ ℝ^{n×d_latent}` (n ≤ N_text)이며, 디코더는  
  ```
  X̂ = f_dec(Z)   ( X̂ ∈ ℝ^{N×d_text} )
  ```  
  로 텍스트 시퀀스를 복원한다(논문 Eq. 2).  
* OCR‑style 학습(텍스트–이미지 쌍)으로 비선형 매핑을 학습하여 **10×~20×** 압축 비율에서도 60 % 이상 정확도 달성.

### 4. 데이터 엔진 & 학습 파이프라인

| 데이터 종류 | 비중 | 주요 내용 |
|-------------|------|------------|
| OCR 1.0 | 30 % | 30 M 페이지(중/영어 25 M) – coarse / fine 레이아웃 주석 |
| OCR 2.0 | 20 % | 차트(10 M), 화학식(5 M), 평면 기하(1 M) 등 복합 이미지 |
| 일반 비전 | 20 % | CLIP‑style 이미지‑텍스트 (LAION 등) |
| 텍스트‑only | 10 % | 길이 8192 토큰, LLM 사전 학습용 |

#### 4.1 학습 단계
1. **DeepEncoder 독립 학습**  
   * 데이터 : OCR 1.0/2.0 + 100 M LAION 샘플  
   * 배치 1280, 시퀀스 길이 4096, LR = 5e‑5, 2 epoch, AdamW + cosine annealing  
2. **DeepSeek‑OCR 전체 학습**  
   * 파이프라인 병렬(PP) 4‑단계 (PP0‑PP3) – PP0/PP1: Encoder, PP2/PP3: Decoder  
   * 20 node × 8 A100‑40G, DP = 40, 전체 배치 640  
   * 멀티모달 LR = 3e‑5, 텍스트‑only 90 B tokens/day, 멀티모달 70 B tokens/day  

> 자세한 학습 세팅은 논문의 § 3.5 에 수록되어 있다.

### 5. 성능 요약 (아키텍처와 직접 연관된 지표)

| 압축 비율 (텍스트 / Vision) | Vision Tokens | OCR 정확도 (Fox Bench) |
|-----------------------------|---------------|------------------------|
| 9 – 10×                     | 64            | 96 % ~ 98 % |
| 10 – 12×                    | 100           | 93 % ~ 97 % |
| 20×                         | 64            | 59 % ~ 60 % |

* OmniDocBench 에서는 **Vision Token 평균 100개**(640×640) 로 GOT‑OCR2.0(256 토큰) 을 크게 앞섰으며, **400 토큰**(1280×1280) 에서는 최신 모델들과 동등한 성능을 기록.  

### 6. 설계 의의

1. **고해상도 → 저활성화** : 윈도우‑Attention + 16× Conv 압축 구조로 GPU 메모리 요구량을 크게 낮춤.  
2. **희소 Vision Token** : 64 ~ 800 토큰 수준에서 다양한 문서 유형을 커버, 토큰‑당 효율이 기존 VLM 대비 **5‑10배** 향상.  
3. **다중 해상도 일괄 지원** : 하나의 모델이 Tiny ~ Gundam‑M까지 6가지 해상도 모드를 동시에 학습·추론 가능.  
4. **MoE 기반 경량 디코더** : 3 B 규모 모델의 성능을 유지하면서 추론 비용을 **~500 M** 파라미터 수준으로 감소.
